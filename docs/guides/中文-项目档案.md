# 钉钉审批套印系统项目档案（中文）

## 一、项目概述

- 项目名称：钉钉审批套印系统
- 形态：Flask + Jinja 模板 + Vite 前端构建
- 核心能力：审批数据读取、签名/附件抽取、PDF 坐标套印、预览打印

---

## 二、当前目录结构（核心）

- `app/`：后端应用代码（蓝图路由 + 业务服务）
- `templates/`：Jinja 页面模板
- `static/dist/`：前端构建产物（由 Vite 输出）
- `frontend/`：前端源码与构建工具链
- `data/process_configs.json`：业务流程配置
- `templates/pdf_templates/`：PDF 模板底图目录
- `static/outputs/`：打印合成输出目录

---

## 三、技术栈

### 后端

- Python
- Flask
- requests
- PyMuPDF (`fitz`)

### 前端

- Vite
- Tailwind CSS（本地构建）
- jQuery + Select2
- PDF.js（`pdfjs-dist`）

---

## 四、业务主链路

1. 门户页获取钉钉授权 code。
2. 后端通过钉钉 API 换取 userid，查询流程实例。
3. 用户点击预览，后端拉取审批详情。
4. 从表单中提取签名/附件 URL。
5. 按 `process_configs.json` 坐标套印到 PDF。
6. 在预览页 PDF.js 中确认并打印。

---

## 五、安全与配置策略

- 已启用环境变量完整性校验：
  - `DINGTALK_APP_KEY`
  - `DINGTALK_APP_SECRET`
  - `DINGTALK_CORP_ID`
  - `FLASK_SECRET_KEY`
- 配置入口：`.env.example`（模板）
- 风险控制：
  - 不再在源码内保留真实默认密钥
  - 业务配置从 `data/process_configs.json` 读取
  - 构建产物和源码分离（`frontend` -> `static/dist`）

---

## 六、维护规范（建议）

1. 禁止将真实密钥写入仓库。
2. 每次改前端后执行：
   - `npm run lint`
   - `npm run build`
3. 变更坐标模板前，先备份 `data/process_configs.json`。
4. 对外部署前，确认 `.env` 权限最小化（仅运行用户可读）。
