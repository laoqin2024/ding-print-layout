## Print & Preview Flow

### Legacy System (套印模块):
1. User clicks "Print" in DingTalk approval
2. Request → /print_view?id={instance_id}&p_code={process_code}
3. Backend checks if designer layout exists
4. If NO designer layout:
   - Load process config from data/process_configs.json
   - Download PDF attachment from DingTalk (if available)
   - Use pdf_service.py to overlay signatures/stamps on PDF template
   - Generate final PDF in static/outputs/
   - Render templates/print.html with PDF URL
5. Frontend (print.js):
   - Uses PDF.js to render PDF pages as canvas
   - Implements dd.biz.util.print() for DingTalk native print
   - Falls back to window.print() if DingTalk API unavailable

### Designer System (新设计器):
1. User clicks "Print" in DingTalk approval
2. Request → /print_view?id={instance_id}&p_code={process_code}
3. Backend checks if designer layout exists in data/print_layouts.json
4. If YES designer layout:
   - Redirect to /designer/preview_print
   - Render templates/designer_print.html
5. Frontend:
   - Calls /designer/api/preview (POST) with layout config
   - Backend uses PyMuPDF (fitz) to:
     * Load base PDF template or attachment
     * Overlay text fields, signatures, stamps, QR codes
     * Apply attachment background if configured
   - Returns generated PDF as blob
   - Displays in <embed> tag
   - Print button calls window.print()

## dd.config and dd.ready Implementation

Location: frontend/src/js/print.js (lines 120-140)

```javascript
async function doPrint() {
  const url = boot.pdfUrl;
  try {
    // Check if DingTalk JSAPI is available
    if (window.dd && dd.ready && dd.biz && dd.biz.util && typeof dd.biz.util.print === "function") {
      dd.ready(function () {
        try {
          // Use DingTalk native print API
          dd.biz.util.print({
            url,
            onSuccess: function () {},
            onFail: function () {
              // Fallback to browser print
              window.print();
            },
          });
        } catch {
          window.print();
        }
      });
      return;
    }
  } catch {
    // fallback handled below
  }
  // Final fallback to browser print
  window.print();
}
```

ATTACHMENT BACKGROUND FEATURE (Recently Fixed)
================================================================================

## Problem
DingTalk attachments are returned as JSON with fileId, not direct URLs:
```json
[{
  "uploadSize": 62988,
  "spaceId": "260817343",
  "fileName": "file.pdf",
  "fileSize": 98612,
  "fileType": "pdf",
  "fileId": "219706204543"
}]
```

## Solution
1. Extract fileId from attachment JSON
2. Call DingTalk API: /v1.0/workflow/processInstances/spaces/files/urls/download
3. Get temporary download URL
4. Download PDF and use as background

## Implementation
- app/services/dingtalk_service.py:
  - get_process_attachment_url() - Converts fileId to download URL
  - download_specific_attachment_pdf() - Downloads specific attachment
- app/routes/designer.py:
  - render_attachment_bg() - Renders attachment as PNG for canvas
  - designer_preview() - Uses attachment as PDF background

## Configuration
In data/print_layouts.json:
```json
{
  "PROC-XXX": {
    "attachment_background_config": {
      "enabled": true,
      "field_id": "DDAttachment_1PKSP5YV9WRGG",
      "attachment_index": 0,
      "page_index": 0,
      "apply_mode": "preview_and_print"
    }
  }
}
```
